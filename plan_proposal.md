# План разработки системы «Структура работы приемки»

Исходя из утвержденной архитектуры (7 ИИ-агентов) и визуального пайплайна «Структура работы приемки», я предлагаю разбить разработку на 3 последовательных этапа. Это позволит поставлять функционал итеративно и тестировать каждый модуль отдельно.

## Этап 1: Инфраструктура, Извлечение данных и Классификация (Фундамент)
*Цель: Научить систему принимать документы, распознавать их тип и извлекать базовый текст/метаданные для дальнейшего анализа.*

1. **Разработка сервиса загрузки (API Gateway)**
   - Реализация эндпоинта для приема входящих документов (PDF, DOCX, XML, изображения).
   - Интеграция с подсистемой антивирусной проверки файлов.
   - Построение системы очередей (например, RabbitMQ/Celery) для асинхронной обработки тяжелых файлов.
2. **Создание Агента "Document Analyzer" (Модель разделения типов файлов)**
   - Интеграция OCR-решений (PyMuPDF для текста, Tesseract/LayoutLM для сканов) для извлечения сырого текста.
   - Создание классификатора: распределение файлов на текстовые данные (ПЗ), графику (чертежи), сметы, машиночитаемые (XML).
   - Выделение метаданных: извлечение структурных элементов паспорта объекта, УИН из заявлений.
   - Запуск "Проверки по формальному признаку" (наличие нужных томов, ЭЦП, соблюдение формата имен файлов).

## Этап 2: ИИ-Аналитика, RAG и Бизнес-Логика (Ядро системы)
*Цель: Подключить большие языковые модели для глубокого анализа контента и проверки на соответствие строительным нормам.*

1. **Разработка "Knowledge Base Agent" (Векторная база данных)**
   - Подготовка и очистка нормативной базы (ПП №87, №963, ПП №154, ГОСТы, СП).
   - Настройка Pinecone/Milvus и пайплайна векторизации (embedding models: OpenAI/Cohere).
   - Реализация гибридного поиска (семантика + BM25) для точного извлечения норм.
2. **Создание Агентов "PP87/PP154 Compliance" и интеграция с LLM**
   - Интеграция Оркестратора для маршрутизации проверок (텍ст -> проверка ПЗ, графика -> проверка АР и т.д.).
   - Настройка промпт-шаблонов для LLM: формирование логики "Проверка соответствия информации в документах" (Ground Truth Check).
   - Разработка алгоритмов кросс-валидации (сверка ТЭП в разных файлах).
3. **Реализация "Обработки исключений" (Disagreement Log)**
   - Механизм перехвата файлов/запросов, с которыми LLM или OCR не справились (низкий % уверенности).
   - Создание интерфейса/API для "Human-in-the-Loop Agent" (передача спорных случаев эксперту).

## Этап 3: Формирование результатов, Интеграция и Пользовательский интерфейс
*Цель: Завершить пайплайн формированием итогового заключения и подключением к рабочим местам пользователей.*

1. **Разработка "Report Generator" (Подготовка замечаний)**
   - Создание шаблонизатора (Jinja2/PDF-рендеринг) для сборки выявленных ошибок в сводное экспертное заключение по ГОСТ Р 7.0.97-2016.
   - Автоматическая приоритизация (Критикал, Мажор, Минор) и маппинг на конкретные пункты СП.
2. **Реализация "Прямых запросов пользователей"**
   - Создание чат-интерфейса или API, где пользователь может задать вопрос к конкретному загруженному пакету документации ("почему здесь замечание?", "покажи источник").
3. **E2E (End-to-End) Тестирование пайплайна**
   - Прогон тестовых наборов проектной документации через всю цепочку: `Входящие -> Формальная проверка -> Разбор контента -> LLM анализ -> Отчет`.
   - Настройка CI/CD и развертывание на серверах.

---
**С чего начнем?**
Я рекомендую стартовать с **Этапа 1, пункта 2 (Document Analyzer)**. Мы можем взять несколько примеров PDF/DOCX (ваши Пояснительные записки) и написать скрипт на Python, который научится надежно извлекать из них текст, таблицы и классифицировать по разделам в JSON формат.
