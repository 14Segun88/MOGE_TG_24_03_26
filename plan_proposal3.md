# Технический план разработки системы «Структура работы приемки» (v3)

> Версия 3 — финальный технический план с маппингом чек-листов, приоритетом XML-парсинга, интеграцией НОПРИЗ, конкретными тест-кейсами, **smart-chunking для RAG, полными метриками для Human-in-the-Loop и развёрнутой архитектурой PP154ComplianceAgent**.
> Нормативная база: ПП РФ №963, ПП РФ №154, ФЗ №190, ГОСТ Р 7.0.97-2016, Приказ №783/пр, Приказ №421/пр.

## Этап 1 — Фундамент: XML-парсинг и Классификация файлов

> **Приоритет №1 — XML-парсинг.** Без него вся система мертва. Пункт 1 чек-листа («ПЗ в XML v1.05») это точка входа.

### 1.1 Разработка `DocumentAnalyzerAgent`

**Задачи:**
- [ ] Реализовать `XmlParser` с валидацией по официальной XSD-схеме (версия 1.05)
- [ ] Реализовать `FileClassifier` — разбор пакета документов по типам (PyMuPDF, Tesseract/LayoutLMv3, Camelot, lxml).
- [ ] Реализовать `FormalCheckRunner` — проверки пп. 1, 3, 8, 72, 81, 83, 84. Особое внимание п. 83: реализация проверки наличия математически валидной ЭЦП **ИЛИ** сканированного ИУЛ (Информационно-удостоверяющего листа) через PyMuPDF+Regex.

### 1.2 Сервис загрузки (API Gateway)

**Стек:** FastAPI + синхронная обработка (без очередей на MVP)
- Целевое время полного пайплайна: от 15 до 30 минут.

## Этап 2 — Ядро: RAG, PP963 и Внешние интеграции

### 2.1 Knowledge Base Agent (RAG)

- **Стек:** Pinecone/Weaviate + `text-embedding-3-large` (OpenAI) + `rerank-multilingual-v2.0` (Cohere).
- **Ключевая задача (`KnowledgeBaseIndexer`):** Реализация логики `smart_chunking`. Документы парсятся не по фиксированному количеству токенов, а по семантической структуре: глава -> пункт -> подпункт. Это предотвращает разрыв юридического контекста.

### 2.2 PP963ComplianceAgent

- Выполняет RAG-запросы и LLM-валидацию по пунктам чек-листа ПП №963 (16-20, 24 и др.).
- **Кросс-валидация ТЭП (Ground Truth Check):** Площадь в ПЗ (Раздел 1) vs АР (Раздел 3); этажность; реквизиты ГПЗУ.

### 2.3 External Integration Agent (Интеграция НОПРИЗ)

**Задача:** Автоматически верифицировать пп. 66 и 67 (членство ГИП в НОПРИЗ).

> [!WARNING]
> У НОПРИЗ нет публичного REST API. `https://nrs.nopriz.ru` — HTML-форма. Ниже описана реальная стратегия.

**Основная стратегия:** Playwright-парсер (`playwright.chromium.launch()`) — логин на `nrs.nopriz.ru`, заполнение формы поиска по ФИО + ИНН, парсинг HTTP-ответа. Сесссионные cookie сохраняются в `storage_state.json` для повторных запросов.

**Стратегия деградации при поломке парсера:** Задача автоматически передаётся `HumanInTheLoopAgent` с триггером `is_edge_case`. Эксперт проверяет реестр вручную и логирует результат в `Disagreement Log`.

**Мониторинг:** Добавить тест, который еженедельно проверяет корректность CSS-селекторов на странице НОПРИЗ (автоматический алерт при изменении вёрстки).

## Этап 3 — Масштабирование: Схемы теплоснабжения (ПП №154)

### 3.1 Разработка `PP154ComplianceAgent`

**Это полноценный, сложный домен экспертизы, требующий отдельного спринта:**
- Проверка обязательного наличия **13 разделов** схемы теплоснабжения.
- Математическая валидация баланса тепловой энергии (Производство = Потребление + Потери). Допустимый коридор невязки: **≤ 2%**.
- Проверка горизонта планирования инвестиционной программы: **≥ 15 лет**.
- Контроль наличия файлов "Электронной модели системы теплоснабжения" (валидация специфичных расширений ПО: Zulu Thermo, Пирамида).

## Этап 4 — Результаты: Заключение и Human-in-the-Loop

### 4.1 Report Generator Agent

- **Сборка PDF (reportlab):** Формирование отчёта строго по ГОСТ Р 7.0.97-2016 (шрифты, отступы, нумерация).
- **Двумерная группировка:** 
  1. По разделам (`by_section`: ПЗ, АР, КР, ООС...).
  2. По критичности (`by_severity`: critical / warning / info).

### 4.2 Human-in-the-Loop Agent

**Полная реализация 4-х триггеров перехвата управления:**
1. `critical_error`: Выявлено критическое несоответствие (например, нет УИН или ЭЦП/ИУЛ).
2. `confidence < 0.7`: Уверенность LLM или RAG ниже 70%.
3. `agent_disagreement`: Оркестратор фиксирует конфликт выводов меджу агентами (например, DocumentAnalyzer говорит "ЭЦП есть", а PP963Agent говорит "Документ не подписан").
4. `is_edge_case`: Распознан нестандартный или редкий тип объекта (например, атомная станция).

**Задача:** Логирование ответов эксперта в `Disagreement Log` для дообучения.
