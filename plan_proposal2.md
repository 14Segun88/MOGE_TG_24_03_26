# Технический план разработки системы «Структура работы приемки» (v2)

> Версия 2 — детализированный технический план с маппингом чек-листов, приоритетом XML-парсинга, интеграцией НОПРИЗ и конкретными тест-кейсами.
> Нормативная база: **ПП РФ №963** (заменил №87 с 01.09.2022), **ПП РФ №154**, **ФЗ №190**, **ГОСТ Р 7.0.97-2016**, **Приказ №783/пр**, **Приказ №421/пр**.

## Этап 1 — Фундамент: XML-парсинг и Классификация файлов

> **Приоритет №1 — XML-парсинг.** Без него вся система мертва. Пункт 1 чек-листа («ПЗ в XML v1.05») это точка входа.

### 1.1 Разработка `DocumentAnalyzerAgent`

**Задачи:**
- [ ] Реализовать `XmlParser` с валидацией по официальной XSD-схеме (версия 1.05)
  - Входные данные: XML-файл ПЗ
  - Выход: Python-словарь со структурированными полями (УИН, дата, реквизиты разделов)
- [ ] Реализовать `FileClassifier` — разбор пакета документов по типам:
  - `text` (PDF с текстовым слоем, DOCX) → PyMuPDF
  - `scanned` (PDF без текстового слоя, TIFF) → Tesseract / LayoutLMv3
  - `drawing` (PDF с чертежами, DWG) → GPT-4-Vision
  - `spreadsheet` (XLSX, PDF со сметами) → Camelot / Tabula
  - `machinereadable` (XML v1.05, JSON) → lxml / json
- [ ] Реализовать `FormalCheckRunner` — проверки пп. 1, 3, 8, 72, 81, 83, 84:
  - Наличие обязательных файлов в пакете
  - Regex-проверка имён файлов (Приказ №783/пр)
  - Проверка наличия и формата ЭЦП

**JSON-интерфейс выхода агента:**
```json
{
  "document_id": "uuid",
  "classification": {
    "xml_pz": true,
    "xml_version": "1.05",
    "files": [{"name": "R1_PZ.xml", "type": "machinereadable"}, ...]
  },
  "formal_checks": {
    "1": {"status": "PASS", "detail": "XML v1.05 найден"},
    "8": {"status": "FAIL", "detail": "УИН отсутствует в заявлении"},
    "83": {"status": "PASS", "detail": "3 ЭЦП найдено"}
  },
  "raw_text": "...",
  "metadata": {"uin": "...", "zastroyschik": "...", "gip_fio": "...", "gip_inn": "..."}
}
```

### 1.2 Сервис загрузки (API Gateway)

**Стек:** FastAPI + синхронная обработка (без очередей на MVP)

```
POST /api/v1/submit → принимает ZIP с пакетом ПД
GET  /api/v1/status/{doc_id} → статус обработки
GET  /api/v1/report/{doc_id} → итоговое заключение
```

## Этап 2 — Ядро: RAG, Агенты соответствия и Внешние интеграции

### 2.1 Knowledge Base Agent (RAG)

- **Стек:** Pinecone/Weaviate + `text-embedding-3-large` (OpenAI) + Cohere Rerank
- **Источники для индексации:** ПП №963 (актуальная редакция), ГрК РФ, ФЗ №190, ФЗ №116, 50+ СП, 30+ ГОСТ
- **Размер базы:** 500 000+ чанков (чанк = 512 токенов с перекрытием 50 токенов)
- **Запрос:** гибридный поиск BM25 (точный) + cosine similarity (семантический), Reranker на топ-20 → топ-5

### 2.2 PP963ComplianceAgent

- Принимает JSON от `DocumentAnalyzerAgent`
- Для каждого пункта чек-листа (16-20, 24-28...) формирует RAG-запрос → получает норму из КБ → через LLM (GPT-4 Turbo / Claude 3.5) верифицирует соответствие
- **Ground Truth Check — кросс-валидация ТЭП:**
  - Общая площадь из XML Раздела 1 (ПЗ) vs. XML Раздела 3 (АР) — допустимое отклонение < 1м²
  - Этажность из ПЗ vs. графической части (АР)
  - Реквизиты ГПЗУ в ПЗ vs. приложенный файл ГПЗУ

### 2.3 External Integration Agent (НОВЫЙ — интеграция НОПРИЗ)

**Задача:** Автоматически верифицировать пп. 66 и 67 (членство ГИП в НОПРИЗ).

**Входные данные:** `gip_fio`, `gip_inn` из JSON DocumentAnalyzerAgent

**Алгоритм:**
1. Запрос к API НОПРИЗ (`https://nrs.nopriz.ru`) по ФИО и ИНН ГИПа
2. Парсинг ответа (статус: «Действующий» / «Исключён»)
3. Запись результата в JSON: `{"66": "PASS/FAIL", "67": "PASS/FAIL"}`

**Резервный вариант (если API нет):** Playwright-парсинг веб-страницы реестра с сохранением сессии.

## Этап 3 — Результаты: Заключение и Интерфейс

### 3.1 ReportGeneratorAgent

- Принимает агрегированные JSON от всех агентов
- Сортирует замечания: **Критичное** (пп. 1, 8, 83 — блокирует экспертизу) / **Мажорное** (пп. 16-20) / **Минорное**
- Генерирует PDF по ГОСТ Р 7.0.97-2016: структура — Вводная часть → Замечания с нумерацией → Рекомендации → Вывод

### 3.2 Human-in-the-Loop Agent

- Активируется при `confidence_score < 0.70` от LLM
- Создаёт задание в интерфейсе эксперта: текст нормы + текст из ПД + вопрос агента
- Ответ эксперта логируется в `Disagreement Log` (JSON-файл) для дальнейшего fine-tuning
