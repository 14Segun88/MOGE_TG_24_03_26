# XML Comparator — сервис сравнения ПЗ и ЗнП

Сервис автоматически сравнивает два XML-документа:
- **ПЗ** — Пояснительная записка (корневой элемент `<ExplanatoryNote>`, схема v01.06)
- **ЗнП** — Задание на проектирование (корневой элемент `<Document>`, схема v01.00)

По результату формируется подробный JSON- и HTML-отчёт: какие поля совпали, какие расходятся, какие отсутствуют. Дополнительно оба файла проверяются по XSD-схемам — результат валидации также включается в ответ.

## Что делает сервис

1. Принимает два XML-файла (ПЗ и ЗнП) и файл маппинга (JSON).
2. Валидирует оба файла по XSD-схемам (независимо от сравнения).
3. Для каждого правила маппинга извлекает значения из обоих документов по XPath.
4. Нормализует значения (убирает лишние пробелы, приводит к нижнему регистру и т.д.).
5. Сравнивает значения выбранной стратегией.
6. Возвращает JSON-отчёт со статусом каждой проверки, значениями из обоих документов, результатом XSD-валидации и пояснениями.
7. Сохраняет все файлы отчёта на диск в одну папку на каждый запрос.

## Быстрый старт — локально (Python)

### Требования

- Python 3.11 или новее
- Windows / Linux / macOS

### Установка

```bash
# 1. Перейдите в папку проекта
cd xml_comparator

# 2. Создайте виртуальное окружение
python -m venv .venv

# 3. Активируйте окружение
# Windows:
.venv\Scripts\activate
# Linux/macOS:
source .venv/bin/activate

# 4. Установите зависимости
pip install -r requirements.txt

# 5. Создайте файл конфигурации
cp .env.example .env
```

### Настройка `.env`

Откройте `.env` и при необходимости укажите путь к файлу маппинга:

```
MAPPING_FILE_PATH=mapping_PZ_ZnP.json
```

Остальные параметры можно оставить по умолчанию.

### Запуск

```bash
python run.py
```

Сервис запустится на `http://localhost:8000`.

Интерактивная документация API: `http://localhost:8000/docs`

## Быстрый старт — Docker

### Требования

- Docker 20.10+
- docker compose v2 (или `docker-compose` v1)

### Сборка и запуск

```bash
# Из папки xml_comparator/
docker compose up --build
```

Или отдельными командами:

```bash
docker build -t xml-comparator .
docker run -p 8000:8000 xml-comparator
```

Сервис будет доступен на `http://localhost:8000`.

> **Примечание:** образ включает XSD-схемы для валидации и файл маппинга. При обновлении схем или маппинга — пересобрать образ (`docker compose up --build`).

### Сохранение отчётов при запуске через Docker

По умолчанию `docker-compose.yml` монтирует папку `./reports` на хосте в `/app/reports` контейнера:

```yaml
volumes:
  - ./reports:/app/reports
```

Каждый запрос создаёт подпапку `reports/<YYYYMMDD>/<uuid>/` со всеми файлами отчёта.

### Использование внешнего файла маппинга

Чтобы использовать внешний файл маппинга вместо встроенного — раскомментируйте секцию в `docker-compose.yml`:

```yaml
volumes:
  - ./mapping_PZ_ZnP.json:/app/mapping_PZ_ZnP.json:ro
```

## API — эндпоинты

Базовый URL: `http://localhost:8000/api/v1`

### `GET /health`
Проверка работоспособности сервиса.

```bash
curl http://localhost:8000/api/v1/health
# {"status": "ok", "version": "1.0.0"}
```

---

### `POST /compare`
Основной эндпоинт. Принимает три файла через `multipart/form-data`.

| Поле | Тип | Описание |
|------|-----|----------|
| `file_pz` | XML-файл | Пояснительная записка |
| `file_znp` | XML-файл | Задание на проектирование |
| `file_mapping` | JSON-файл | Файл маппинга |

```bash
curl -X POST http://localhost:8000/api/v1/compare \
  -F "file_pz=@pz.xml" \
  -F "file_znp=@znp.xml" \
  -F "file_mapping=@mapping_PZ_ZnP.json"
```

---

### `POST /compare/preset`
Сравнение с маппингом, хранящимся на сервере (из `MAPPING_FILE_PATH` в `.env`).

| Поле | Тип | Описание |
|------|-----|----------|
| `file_pz` | XML-файл | Пояснительная записка |
| `file_znp` | XML-файл | Задание на проектирование |
| `mapping_path` | query-параметр (необязательно) | Путь к маппингу на сервере |

```bash
curl -X POST http://localhost:8000/api/v1/compare/preset \
  -F "file_pz=@pz.xml" \
  -F "file_znp=@znp.xml"
```

---

### `GET /mapping/rules`
Список правил из файла маппинга с фильтрацией.

```bash
curl "http://localhost:8000/api/v1/mapping/rules?mapping_path=mapping_PZ_ZnP.json&section=Объект&risk=Низкий"
```

## Формат JSON-отчёта

Ответ содержит два объекта — `report` (сравнение по маппингу) и `xsd_validation` (результат XSD-валидации):

```json
{
  "report": {
    "metadata": {
      "generated_at": "2025-01-01T12:00:00Z",
      "mapping_file": "mapping_PZ_ZnP.json",
      "mapping_rules_total": 215,
      "document_pz": { "file_name": "pz.xml", "schema_version": "01.06" },
      "document_znp": { "file_name": "znp.xml" },
      "comparator_version": "1.0.0"
    },
    "summary": {
      "total_rules": 215,
      "total_compared": 80,
      "total_skipped": 135,
      "success_count": 65,
      "failed_count": 10,
      "missing_in_pz_count": 2,
      "missing_in_znp_count": 3,
      "missing_in_both_count": 0,
      "error_count": 0
    },
    "checks": [
      {
        "rule_id": "R001",
        "section": "Объект",
        "subsection": "Наименование",
        "status": "failed",
        "source_value_pz": "Жилой дом",
        "source_value_znp": "Жилой дом №5",
        "expected_value": "Жилой дом №5",
        "message": "Расхождение: ПЗ содержит «Жилой дом», а ЗнП — «Жилой дом №5»"
      }
    ]
  },
  "xsd_validation": {
    "pz_valid": true,
    "znp_valid": false,
    "pz": { "file": "pz.xml", "is_valid": true, "error_count": 0, "errors": [] },
    "znp": { "file": "znp.xml", "is_valid": false, "error_count": 2, "errors": [
      { "line": 45, "column": 0, "level": "error", "message": "..." }
    ]}
  }
}
```

### Поле `expected_value`

Поле `expected_value` в каждой проверке содержит **ожидаемое значение из ЗнП** (задание на проектирование):
- при `status: failed` — нормализованное значение ЗнП, которому должно соответствовать ПЗ
- при `status: missing_pz` — значение из ЗнП, которое отсутствует в ПЗ
- при `status: missing_znp` — `null` (ЗнП не содержит значения)

## XSD-валидация

При каждом запросе оба XML-файла автоматически проверяются по XSD-схемам:

- ПЗ → `xsd and xsl/XSD Пояснительная записка версия 01.06/explanatorynote-01-06.xsd`
- ЗнП → `xsd and xsl/XSD Задание на проектирование_01-00/DesignAssignment-01-00.xsd`

Схемы используют стандарт **XSD 1.1** (библиотека `xmlschema`).

**Важно:** XSD-валидация и сравнение по маппингу — **независимые процессы**. Даже если файл не прошёл XSD-валидацию, сравнение по маппингу всё равно выполняется и включается в ответ. Результат валидации находится в поле `xsd_validation` ответа.

## Сохранение отчётов на диск

При каждом запросе к `/compare` или `/compare/preset` все файлы сохраняются в одну папку:

```
reports/
└── 20250322/                        ← дата запроса (YYYYMMDD)
    └── 3f7a1b2c-4d5e-6f7a-8b9c-.../  ← уникальный UUID
        ├── report.json              ← JSON-отчёт сравнения по маппингу
        ├── report.html              ← HTML-отчёт с визуализацией ПЗ
        ├── validation.json          ← JSON-отчёт XSD-валидации
        ├── validation.html          ← HTML-отчёт XSD-валидации
        ├── pz.xml                   ← загруженный файл ПЗ
        └── znp.xml                  ← загруженный файл ЗнП
```

- Если `REPORTS_DIR` в `.env` пустой — сохранение отключено, ответ возвращается только в HTTP.
- Ошибка записи на диск **не прерывает** запрос.

## Файл маппинга

Файл `mapping_PZ_ZnP.json` содержит правила сравнения полей ПЗ и ЗнП.

### Структура файла

```json
{
  "mapping_rows": [
    {
      "section": "Объект",
      "subsection": "Наименование объекта",
      "label_pz": "Наименование объекта (ПЗ)",
      "xpath_pz": "/ExplanatoryNote/{ТипОбъекта}/ObjectName",
      "label_znp": "Наименование объекта (ЗнП)",
      "xpath_znp": "/Document/Content/Object/ObjectName",
      "compare_mode": "Среднее",
      "risk": "Низкий",
      "reason": "Должны совпадать"
    }
  ]
}
```

### Плейсхолдер `{ТипОбъекта}`

В XPath для ПЗ используется плейсхолдер `{ТипОбъекта}`. Движок автоматически подставляет:
- `NonIndustrialObject` — непроизводственный объект
- `IndustrialObject` — производственный объект
- `LinearObject` — линейный объект

Тип определяется по наличию соответствующего дочернего элемента в `<ExplanatoryNote>`.

## Режимы сравнения

| Режим | Описание |
|-------|----------|
| **Жёсткое** | Trim + Unicode NFC + lowercase. Значения должны совпадать побуквенно. |
| **Среднее** | + нормализация кавычек, тире, ОПФ, пробелов. Покрывает типичные расхождения оформления. |
| **Мягкое** | Значения со словом «не» как отдельным словом считаются семантически пустыми. Если оба поля непустые — проверяется факт наличия. |
| **Булево-мягкое** | ЗнП содержит `true`/`false`. `false` → ПЗ пусто или содержит отрицание. `true` → ПЗ непусто без отрицания. |
| **Фиксированное** | Каждый документ проверяется на своё ожидаемое значение из `type_pz`/`type_znp`. |

### Уровни риска

| Риск | Стратегия |
|------|-----------|
| **Низкий** | StrictScalar — расхождение всегда ошибка |
| **Средний** | MediumScalar — расширенная нормализация перед сравнением |
| **Высокий** | Информационно — форматы принципиально различаются |

## Статусы проверок

| Статус | Отображение | Описание |
|--------|-------------|----------|
| `success` | ✓ Совпадает | Значения совпадают после нормализации |
| `failed` | ✕ Расхождение | Значения не совпадают |
| `missing_pz` | ⚠ Отсутствует в ПЗ | XPath ПЗ не нашёл значение |
| `missing_znp` | ○ Отсутствует в ЗнП | XPath ЗнП не нашёл значение |
| `missing_both` | Нет в обоих | Оба XPath вернули пустой результат |
| `skipped` | Пропущено | Правило пропущено (нет XPath) |
| `only_pz` | Только в ПЗ | Правило только для ПЗ, в ЗнП нет аналога |
| `error` | Ошибка | Внутренняя ошибка при обработке правила |

## Сопоставление списков документов («2 из 3»)

В разделах **Документы-основания** и **ИРД** сервис сопоставляет элементы ПЗ и ЗнП попарно — **не по порядку следования**, а по содержанию.

### Алгоритм сопоставления

Каждый элемент идентифицируется тремя критериями:

| # | Критерий | Поле ПЗ | Поле ЗнП |
|---|----------|---------|---------|
| 1 | Тип документа | `DocType` | `@Type` |
| 2 | Имя файла | `File/FileName` | `File/Name` |
| 3 | Контрольная сумма | `File/FileChecksum` | `File/Checksum` |

**Правило «2 из 3»:** совпало ≥ 2 критерия → документы сопоставлены, поля сравниваются; совпало < 2 → статус `missing_pz` / `missing_znp`.

## Специальные правила

### R035 — Постоянное пребывание людей

В ЗнП — булев атрибут (`true`/`false`), в ПЗ — текстовое описание. Режим `Булево-мягкое`:

| ЗнП | ПЗ | Результат |
|-----|----|-----------|
| `false` | Пусто или «Не предусмотрено» | ✓ success |
| `false` | «Предусмотрено проживание» | ✗ failed |
| `true` | «Постоянное проживание — 50 чел.» | ✓ success |
| `true` | Пусто или «Не предусмотрено» | ✗ failed |

## Разработка и расширение

Подробная архитектурная документация — в файле [ARCHITECTURE.md](ARCHITECTURE.md).

```
HTTP-запрос → FastAPI (router.py)
    → make_report_dir() — создаётся одна папка для всех файлов запроса
    → XSD-валидация (xsd_validator.py) → validation.json / validation.html
    → Загрузчик маппинга (loader.py)
    → XML-парсер (xml_parser.py)
    → Движок сравнения (comparator.py)
        → Реестр стратегий (registry.py)
            → StrictScalarStrategy / MediumScalarStrategy
                → Нормализаторы (normalizers/)
    → Построитель отчёта (builder.py)
    → HTML-отчёт (html_report.py) → report.html
    → storage.save_report() → report.json, pz.xml, znp.xml
    → JSON-ответ (CompareResponse)
```
