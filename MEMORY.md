# MEMORY — МособлГосЭкспертиза DocumentAnalyzer Bot

## Контекст проекта
- **Проект**: Автоматизированная система экспертизы строительной документации (МособлГосЭкспертиза)
- **Цель**: 98% точности через специализацию агентов + накопительный HITL
- **Железо**: GTX 1650 (4 ГБ VRAM), 16 ГБ RAM, WSL 2 Ubuntu 22.04
- **Стек**: Python 3.10, FastAPI, Telegram Bot (python-telegram-bot), SQLAlchemy, Weaviate (Docker), Playwright
- **RAG**: Weaviate 1.28.4, 469 чанков, BM25 + sentence-transformers (CrossEncoder отключён из-за SIGSEGV)
- **LLM**: Groq API, 3 аккаунта Round-Robin (90 req/min суммарно)
- **Запуск**: `./start.sh` (Weaviate → LM Studio check → bot.py)
- **Папка**: `~/Практика в машинном обучении/`
- **Зависимости**: `pip install -r requirements.txt && playwright install chromium`

## Ошибки (не повторять!)
- **Playwright sync в asyncio**: падает с "Sync API inside asyncio loop". Решение: `asyncio.to_thread(agent.method, args)`
- **CrossEncoder SIGSEGV**: загрузка CrossEncoder(PyTorch) внутри asyncio → SIGSEGV. Отключён, используем BM25.
- **NOPRIZ selectors**: старые `input[name='snils']` не существуют. Реальный UI — таблица с фильтрами, 2-й input = ФИО, кнопка "Применить"
- **storage_state.json 0 cookies**: нормально — сайт не использует cookies для сессии
- **WSL 2**: `nohup` не спасает от перезапуска WSL; пути с пробелами в кавычках
- **PowerShell**: `curl` = Invoke-WebRequest (не curl!), использовать `curl.exe`
- **zakonrf.info**: не покрывает ФЗ-214, ФЗ-127. Использовать consultant.ru
- **CNTD Unstable IDs** [25.03.2026]: Идентификаторы на `docs.cntd.ru` могут меняться или вести на судебные акты (SudRF). Например, SP-54.13330.2022 часто подменяется решением суда. Требуется ручная проверка ID через поиск по тексту.
- **Pyre2 lint errors**: "Could not find import" с кириллическими путями — баг Pyre2, не мешает работе бота

### Ошибка: Сообщения > 4096 символов не отправлялись
*   **Симптом:** При отправке длинного текстового отчета бот ловил 400 Bad Request от Telegram (Message is too long), но `error_handler` классифицировал это как сетевую ошибку и пытался продолжить работу, из-за чего итоговое PDF-заключение не отправлялось.
*   **Решение:** Написана helper-функция `_send_long()` в `bot.py`, которая бьет текст на чанки по 4000 символов с учетом переносов строк. В `error_handler` добавлена проверка `isinstance(err, BadRequest)` и обработка "message is too long" как логической ошибки, чтобы не маскировать проблему.

### Ошибка: RemoteProtocolError при загрузке 20+ МБ файлов
*   **Симптом:** При отправке `iLoveZIP_part2.zip` (~18 МБ) Telegram API разрывал соединение `httpx.RemoteProtocolError: peer closed connection`. Это происходило из-за внутренних таймаутов Telegram API и нестабильной сети.
*   **Решение:** 
    1. Увеличены таймауты в `HTTPXRequest` (bot.py параметр `read_timeout=180` вместо `60`).
    2. Скачивание файла (`tg_file.download_to_memory`) обёрнуто в цикл `try...except` на 3 попытки (retry). Если API отрубает соединение, бот ждёт 2 секунды и пробует загрузить эту же часть снова, вместо того чтобы падать с ошибкой.

### Ошибка: Пустые пункты в списке нарушений Сверки ТЗ/ПЗ
*   **Симптом:** В итоговом сообщении в блоке "Пропущено" выводилось пустое тире (например `⚠️ -`), потому что в исходной таблице ТЗ/ПЗ были пустые ячейки (пробелы, тире, "Нет").
*   **Решение:** В `bot.py` добавлена функция `is_valid_req(req)` при формировании списков `violations` и `skipped`, которая отсеивает строки короче 3 символов, дефисы и слова "нет".

### Ошибка: LLM падает с 403 Forbidden при кросс-валидации ТЭП
*   **Симптом:** Если Groq API возвращает ошибку (лимиты или бан), бот выводил `❌ Кросс-валидация ТЭП: расхождения!` и тех. текст `Error code: 403 - Forbidden`. Пользователь думал, что ТЭП не совпал, хотя на самом деле отвалился запрос к нейросети.
*   **Решение:** В `bot.py` добавлена проверка на наличие слов "Error", "403" и "Forbidden" в массиве `tep_discrepancies`. Теперь при падении API выводится статус `⚠️ Кросс-валидация ТЭП: ошибка LLM API!`.

### Однообразный вывод: Неизвестные форматы в Telegram
*   **Симптом:** Файлы с расширениями `.sig` и `.xml` (кроме пояснительной записки) выводились в телеграмме с иконкой "❓ Неизвестный" или просто "❓ ТЗ".
*   **Решение:** В `src/agents/document_analyzer/file_classifier.py` добавлен `FileType.SIG` для классификации подписей. В `bot.py` добавлен полноценный словарь маппингов `file_type_icons`, который задает красивые иконки для `sig` (🔐 ЭЦП), `xml_other` (📄 XML), `estimate` (💰 Смета), `drawing` (📐 Чертёж) и `archive` (📦 Архив).

### Ошибка: ValueError: negative seek value при чтении ZIP-архива
*   **Симптом:** При загрузке больших архивов, особенно тех, которые были созданы с использованием ZIP64 (или разбиты на части), бот крашился с ошибкой `ValueError: negative seek value ...` внутри библиотеки `zipfile.py`. В логах: `zf.read(zinfo.filename) ... _file.seek(self._pos) ValueError: negative seek value`.
*   **Причина:** Прямое скачивание больших ZIP-файлов в буфер оперативной памяти (`io.BytesIO()`) вызывает сбой позиционирования указателя в `zipfile`, если внутри архива есть файлы с большими смещениями (ZIP64), так как TG API мог отдавать куски с некорректными мета-блоками при потоковой записи.
*   **Решение:** В `bot.py` я заменил `io.BytesIO` на временный физический файл `tempfile.NamedTemporaryFile`. Теперь файл полностью докачивается на диск (`download_to_drive`), и только после этого открывается через `zipfile.ZipFile(tmp_path, "r")`. Код обёрнут двойным `try...finally` для гарантированного удаления `tmp_path` с диска (защита от утечек места).

### Ошибка: OSError: [Errno 22] Invalid argument при zf.read(filename)
*   **Симптом:** При попытке прочитать файл из загруженного скачанного ZIP архива (`zf.read()`) Python падал с сообщением об ошибке ОС: Invalid argument в вызове `_file.seek()`.
*   **Причина:** Даже локально скачанные файлы могут вызвать баг в `zipfile`, если они имеют сложную структуру сжатия, разбиты архиватором 7zip, или используют ZIP64 с огромными смещениями. В таком режиме метод `zf.read()` ломается при попытке рассчитать и применить `seek()` к внутреннему файловому указателю.
*   **Решение:** Отказ от `zf.read(filename)` внутри памяти в `bot.py`. Вместо него был реализован метод извлечения `zf.extract(zinfo, path=extract_dir)` во временную папку. Только после того, как файл физически записан на диск, он считывается стандартным питоновским `open(..., "rb").read()` и тут же удаляется. Это 100% надёжный способ чтения файлов из проблемных архивов.

### Ограничение 50МБ от Telegram API 
*   **Симптом:** Bot API не принимает файлы (монолитные ZIP) больше 50 МБ.
*   **Причина:** Жесткое ограничение Telegram для ботов. Многотомные ZIP загружать нельзя из-за бага Python `zipfile`, который не умеет склеивать многотомные архивы 7-zip.
*   **Решение:** Добавлена спец. команда администратора `/local_zip <путь>`, которая позволяет боту парсить огромные (300MB+) архивы прямо с файловой системы сервера, обходя Telegram API и физически считывая их во внутреннюю "корзину" сессии бота.
